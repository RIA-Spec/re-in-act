# RAS Concepts

If the introduction explains the idea, this page explains the working parts. Think of the RAS as a small workshop attached to the agent: `reason()` makes local judgments, and `act()` touches the outside world.

This page walks through the two RAS runtime flavors and the two core interfaces with annotated examples drawn from the [one-agent](https://github.com/RIA-Spec/one-agent) reference implementation.

## Control View

Re in Act is easiest to understand as a local control architecture.

- The **top-level reasoner** defines the RAS and its constraints.
- The RAS runs deterministic control flow such as loops, retries, and tool calls.
- `reason()` plays the role of a **local comparator and controller**: its prompt carries the goal, local observations, relevant context, and governing constraints, then it compares local reality against that target and returns schema-bounded output that execution can act on.

This is an engineering analogy, not a claim that the runtime is literally a classical control block diagram. The point is that raw observations become useful only after local filtering and comparison.

If you want the deeper design rationale, see [Control-Theoretic View](/docs/learn/control-theoretic-view).

In practical agent terms, this is why the RAS matters: it lets execution keep working through local uncertainty instead of handing every small disturbance back to the main loop.

## The Two Runtime Flavors

### Code RAS

The agent writes scripts (Python or TypeScript). `reason()` and `act()` are async interfaces available in scope.

```python
import asyncio

async def main():
    # act() — user-defined or provided by the runtime
    m = await act("__manual__", {})

    # reason() — structured output grounded in the tool list
    r = await reason(
        f"Goal: web scraping. Pick relevant tools from: {m['content'][0]['text']}",
        example=["tool_name"]
    )

    for tool in r["data"]:
        info = await act("__manual__", {"name": tool})
        print(info["content"][0]["text"])

asyncio.run(main())
```

**Pattern**: `act()` for data retrieval → `reason()` for selection → loop to `act()` for each result.

The agent can write these scripts interchangeably in **Python** or **TypeScript/Node.js**, since both provide native `async`/`await` control flow.

```typescript
async function main() {
  // act() — user-defined or provided by the runtime
  const m = await act("__manual__", {});

  // reason() — structured output
  const r = await reason(`Goal: web scraping. Pick relevant tools from: ${m.content[0].text}`, [
    "tool_name",
  ]);

  for (const tool of r.data) {
    const info = await act("__manual__", { name: tool });
    console.log(info.content[0].text);
  }
}

main().catch(console.error);
```

**Pattern**: Nearly identical async logic across languages.

### Bash RAS

The agent writes shell pipelines. `reason` and `act` are CLI commands available in `PATH`.

```bash
# Discover and filter tools
act --manual > m.json && \
  cat m.json | reason \
    --prompt "Extract relevant tool names as JSON array:" \
    --prompt - \
    --structure '["bash"]' > tools.json && \
  cat tools.json | jq -r '.data[]' | while read -r t; do
    act --manual "$t"
  done
```

**Pattern**: redirect intermediate results with `>` → use `jq` to extract from `.data` → pipe into the next command.

Unix pipelines are useful here because they compose programs through standard streams while hiding internal details behind clean interfaces. They do not automatically provide semantic understanding; `reason()` is still what turns raw output into a useful structured judgment.

## `reason(prompt, example_output)`

`reason()` is the **only required interface** in the spec. It converts a natural-language request into deterministic structured JSON.

### Key rules

- **Uses a fresh inference context** — each call should include the needed local context explicitly, but the surrounding execution state and orchestration still stay in the same RAS.
- **Include goal + observation + context + constraints** — prompts should state the target objective, local observations/feedback, task-relevant context, and governing constraints or rules.
- **Retry on schema failure** — validates the output against the schema inferred from `example_output`. Retries up to N times with the error as feedback before returning `{ "error": "..." }`.
- **Bound output variety with schema** — `example_output` constrains responses into an executable control channel.
- **Use a lightweight model when possible** — `reason()` is an atomic local reasoning call, not a full top-level reasoning cycle.

### What `reason()` is for

`reason()` is not a second full agent. It is a local judgment step inside an orchestrable RAS. Its job is to compare goal against local reality, use the resulting deviation to decide what matters, and return bounded decisions that execution can carry out deterministically under the control of the enclosing runtime.

That is why `reason()` improves execution quality: it lets the agent compress noisy local observations and intermediate data into something small, explicit, and actionable.

Conceptually, `reason()` includes two tightly coupled steps:

- a **comparison step** that assesses the relevant gap between the goal and the current local reality
- a **control step** that turns that gap into a bounded output that the runtime can use immediately

Typical uses include:

- classify success or failure from noisy logs
- pick the next tool or branch from a constrained set
- extract structured facts from unstructured local evidence
- summarize only the signal that should leave the RAS

### Example — sentiment analysis

```python
items = ["Great product!", "Broke after two days", "Absolutely love it"]
r = await reason(
  f"Goal: classify sentiment. Observation: {items}. Constraints: return one label per item.",
    example=[{"text": "", "sentiment": ""}]
)
for x in r["data"]:
    print(f"{x['text']} → {x['sentiment']}")
```

### Example — binary decision

```python
r = await reason(
  "Goal: decide whether to alert. Observation: 15 errors in the last hour; threshold is 10. Constraints: return a boolean only.",
    example=True
)
if r["data"]:
    print("Alert!")
```

## `act(name, args)` — Optional

`act()` is a **convenience interface** provided by the RAS for calling external tools. The spec does not require you to use it — you may execute actions any way you like.

### When to use `act()`

- Calling MCP tools
- Running bash commands
- Fetching external resources
- Any stateless external operation

### Error handling

```python
result = await act("websearch", {"query": "Re in Act spec"})
if result.get("isError"):
    print(result)
    return
# Safe to use result["content"][0]["text"]
```

### Batch actions

Always batch all related `act()` calls into a single script. Do not split related actions across multiple conversation turns:

```python
# Good — one script, multiple acts
urls = (await reason(f"Top 2 URLs from: {search_result}", ["url1", "url2"]))["data"]
data = [
    (await act("webfetch", {"url": u}))["content"][0]["text"]
    for u in urls
]
summary = (await reason(f"Synthesize: {data}", {"summary": ""}))["data"]["summary"]
print(summary)
```
