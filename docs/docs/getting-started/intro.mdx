# What is Re in Act?

**Re in Act** (Reason in Action) is a simple idea: let the agent think globally, but adapt locally.

In traditional **ReAct** (_Reason → Act → Observe → Repeat_), every small issue has to travel back through the outer loop. That makes agents slow, noisy, and fragile.

**Re in Act** changes this by introducing a **Reason-able Action Space (RAS)**. Main reasoning sets the objective once, and the RAS handles local adjustment during execution.

---

## The Cerebellum Metaphor

In biology, the **Cerebrum** (Big Brain) handles slow, deliberate, and conscious thought. The **Cerebellum** (Small Brain) handles motor control and reflexes: it is fast, continuous, and automatic.

If you decide to walk, your Cerebrum issues the high-level command: "Walk forward." It doesn't calculate every muscle contraction. Instead, your **Cerebellum** takes that command and organizes the complex actions required, dynamically adjusting your balance based on real-time feedback from the terrain.

### Why this matters for AI:

The Cerebellum has two properties that make it the right metaphor for the RAS:

1. **Goal-oriented (fixed target)**: The Cerebellum doesn't perform main reasoning. It receives a high-level command — "walk forward" — and treats it as a fixed reference point. Achieving that goal in the actual environment (balance, terrain, speed) is entirely its own business.
2. **Internal models**: It maintains internal models of the body and environment to dynamically adjust execution in real time, without consulting the Cerebrum for every correction.

Together, these mean the Cerebrum **never needs to enumerate or pre-model all possible disturbances**. It delegates to a system that can absorb local problems where they happen. The theory comes after the intuition:

- **Ashby's Law (Requisite Variety)**: A controller must have at least as much variety as the system it controls. ReAct's fixed Action Primitives lack this variety. The RAS — a programmable, feedback-driven runtime — provides it.
- **Good Regulator Theorem**: Every good regulator must contain an internal model of the system it regulates. The RAS is precisely this: a local model of the tool environment, operating as a **closed-loop feedback system** that self-corrects without escalating every deviation back to main reasoning.

## At a Glance

|                 | Traditional ReAct           | Re in Act                      |
| --------------- | --------------------------- | ------------------------------ |
| **Logic**       | Probabilistic (LLM-stepped) | **Deterministic** (RAS-driven) |
| **Metaphor**    | Micro-managing Cerebrum     | **Autonomous Cerebellum**      |
| **Round Trips** | Continuous outer-loop turns | **One delegated action phase** |
| **Context**     | Grows with observations     | **Isolated from main context** |
| **Resilience**  | Brittle to perturbations    | **Self-correcting via RAS**    |

## Key Concepts

### Reason-able Action Space (RAS)

The RAS is the execution context for the delegated action phase. It provides two core interfaces:

- **`reason(prompt, example_output)`** — calls the LLM for structured JSON output (fresh context, no memory)
- **`act(name, args)`** — calls an external tool and returns its result _(optional — you may use your own action executor)_

### Two Runtime Flavors

- **Bash RAS** — Unix pipelines composing `reason` and `act` as CLI commands
- **Code RAS** — Python or TypeScript scripts with native `async/await` control flow
