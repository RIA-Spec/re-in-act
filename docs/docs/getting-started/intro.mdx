# What is Re in Act?

**Re in Act** is about making execution stronger.

In traditional **ReAct** (_Reason → Act → Observe → Repeat_), small execution problems keep bouncing back to the outer loop. That creates extra round trips, noisy context, and brittle control.

Re in Act introduces a **Reason-able Action Space (RAS)** so those local problems can be handled where they appear. Top-level reasoning defines the RAS and its constraints; execution uses that workspace to inspect feedback, make bounded judgments, and keep moving.

The result is simpler execution: fewer outer-loop turns, less context bloat, and a larger effective action space during execution.

---

## The Cerebellum Metaphor

Think about walking. The conscious brain decides to move, but it does not micromanage every correction. Adjustment happens closer to the body and closer to feedback.

That is the intuition behind the RAS.

Top-level reasoning should define the work and its constraints. The execution layer should handle local disturbances inside that bounded workspace instead of escalating every small issue back to the outer loop.

This metaphor matters for two reasons:

1. **Bounded execution**: the RAS works inside a defined task package rather than inventing a new top-level objective.
2. **Local adaptation**: the RAS can use nearby observations, intermediate data, and runtime control flow to adjust while execution is already underway.

That is why the deeper theory points in the same direction:

- **Ashby's Law (Requisite Variety)**: only variety can absorb variety. ReAct's fixed action layer often cannot express enough differentiated responses to handle local disturbances well. The RAS expands that local controller variety.
- **Good Regulator Theorem**: a successful regulator needs access to a model of the system it regulates. In Re in Act, the RAS supplies task-relevant local structure, so the agent does not need to rebuild that structure from scratch on every outer-loop turn.

If you want the deeper version of this argument, see [Control-Theoretic View](/docs/learn/control-theoretic-view).

| **Resilience**  | Brittle to perturbations    | **Self-correcting via RAS**          |

## Key Concepts

### Reason-able Action Space (RAS)

The RAS is the execution context that top-level reasoning defines for an orchestrated action phase. It contains:

- **Team Context** — the work specification, local inputs, and constraints that define the RAS
- **`reason(prompt, example_output)`** — a local semantic judgment step that turns context plus feedback into structured output
- deterministic control flow for retries, branching, and intermediate-data handling

The important point is that the RAS is not just a place to run tools. It is the part of the system that top-level reasoning can still structure and control while execution inspects local observations and intermediate data, makes bounded judgments, and keeps moving without turning every small issue into another outer-loop turn.

It may also provide:

- **`act(name, args)`** — calls an external tool and returns its result _(optional — you may use your own action executor)_

### Two Runtime Flavors

- **Bash RAS** — Unix pipelines composing `reason` and `act` as CLI commands
- **Code RAS** — Python or TypeScript scripts with native `async/await` control flow
