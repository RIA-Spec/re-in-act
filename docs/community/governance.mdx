# Governance

This document describes how the Re in Act project makes decisions about specification and documentation changes.

## Decision-Making

Changes are made through a proposal-first, consensus-seeking process:

1. **Proposal** — A contributor submits a formal proposal
2. **Discussion** — The community reviews and provides feedback
3. **Revision** — The author updates the proposal based on feedback
4. **Decision** — Maintainers accept, defer, or reject with rationale

## Roles

| Role            | Responsibilities                                                          |
| --------------- | ------------------------------------------------------------------------- |
| **Contributor** | Submit issues, proposals, and pull requests                               |
| **Reviewer**    | Review changes for clarity, consistency, and technical soundness          |
| **Maintainer**  | Merge approved changes, publish releases, and resolve tie-break decisions |

## Versioning

Specification versions are released on a date-based schedule (`YYYY-MM-DD`). Each release is immutable once published.

Draft updates continue under `specification/draft/` until promoted to a dated version.
