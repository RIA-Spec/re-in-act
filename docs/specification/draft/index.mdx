## 1. Introduction

This draft proposes **Re in Act** (Reason in Action), a new paradigm and open specification for AI agents that expands the effective action space available during execution.

Instead of forcing main reasoning to micromanage every reason-act-observe turn, Re in Act delegates execution into a **Reason-able Action Space (RAS)** where context-aware `reason()` calls happen **inside** the action phase. This shifts local adaptation from the outer loop into the execution layer itself.

The central advantage is cybernetic, not only operational: ReAct's fixed action interface often lacks requisite variety, so main reasoning must continuously compensate. RAS increases local execution variety and embeds feedback close to action, aligning with Ashby's Law and the Good Regulator principle. Reference implementations show this also collapses many outer-loop iterations into a single delegated action phase, reducing wall-clock latency and main-context growth.

## 2. Motivation

AI agents, especially coding agents, have evolved significantly since the inception of ReAct, which revolutionized problem-solving by interleaving thought and action. Yet fundamental limitations persist.

ReAct relies on iterative cycles of **serialized reasoning and acting**, where execution must pause for observation before the next decision. This architecture suffers from three structural problems and one control-theoretic bottleneck:

### 2.1 Unnecessary Round Trips (High Latency)

Each observation is pushed back into model context, requiring a full outer-loop turn. This "round-trip tax" makes agents feel slow and expensive.

### 2.2 Context Window Pollution

By observing every intermediate result verbatim (e.g., massive build logs), the main context window becomes bloated with noisy, irrelevant data. This leads to attention loss and repetitive loops. The cumulative attention cost scales as $O(L^2)$ with context length $L$ — a fundamental constraint of transformer-based ReAct that compounds with every added round trip.

### 2.3 Fragile Probabilistic Control Flow

The LLM controls the flow at every micro-step. Logic that should be **deterministic** (if/else, loops) is instead **probabilistic**, making the agent's behavior increasingly unreliable as complexity grows.

Concretely, an LLM-controlled `for` or `while` loop has no formal correctness or termination guarantee. Under attention pressure, the model may silently mis-count iterations, skip conditions, or misread loop state — turning what should be deterministic execution into an unreliable probabilistic gamble.

### 2.4 Control Theory

According to Ashby's Law, a controller must have at least as much variety as the system it is intended to control. In ReAct, the practical bottleneck is not only the LLM itself, but the narrow action interface between action and a complex environment: the action layer cannot carry enough variety to absorb environmental perturbations, so required variety is not effectively transmitted where regulation happens.

"Every good regulator of a system must be a model of that system." ReAct therefore forces a global model (the LLM) to compensate continuously for missing local structure. This is an **internal variety tax**: token-intensive repair work, added latency, and rising drift risk as corrective chains grow longer.

Re in Act addresses this by introducing the **Reason-able Action Space (RAS)** as a local, feedback-capable intermediary between main reasoning and the environment. RAS increases local execution variety and provides a task-level internal model at the action layer, so adaptation occurs where action actually happens instead of being repeatedly escalated to main reasoning.

## 3. Overview

Re in Act transforms a serial "Reason and Act" loop into an atomic **"Reason in Action"** primitive: combining deterministic control flow with non-deterministic decisions in one action.

**Benefits:**

- **Fewer round trips**: Multiple outer-loop turns collapse into a single delegated action phase, reducing latency and cost.
- **Clean context window**: Intermediate data (build logs, API responses, intermediate reasoning) stays **inside** the RAS and never enters the main context. Only the final observation is returned.
- **Deterministic control flow**: `for`/`while`/`if` logic runs in code, not through the LLM — correctness is guaranteed by the runtime, not by model attention.
- **Requisite variety**: The RAS is a programmable, adaptive execution layer with the variety needed to handle unmodeled environmental perturbations.

**New complexity introduced:**

A **Reason-able Action Space (RAS)** is required: an execution context that can run a delegated action phase through `reason()` and optional `act()` calls. Main reasoning must be able to generate well-structured delegated programs, and intermediate execution state is not directly observable from the outer loop.

## 4. Specification

### 4.1 Reason-able Action Space (RAS)

The RAS defines the execution context and interfaces for reasoning in action. It can be run in two main forms:

#### Bash RAS — The Unix Philosophy

The agent controls flow using Unix pipes (`|`) and redirection (`>`). Intermediate data is persisted transparently on the filesystem.

```bash
cat api_docs.md | \
  reason \
    --prompt "Read this API documentation:" \
    --prompt - \
    --prompt "Generate a bash command to test this API." \
    --structure '{"cmd": ""}' | \
  jq -r '.data.cmd' | \
  act bash -
```

This pipeline streams API documentation into the agent to generate an executable command, which is then extracted by `jq` and immediately run via `act`.

#### Code Interpreter RAS — The Programmatic Approach

The agent manages control flow using code execution (conditions, loops, branches). Python and TypeScript are recommended to leverage model pre-training on their syntax and async semantics.

```python
log = open("build.log").read()
analysis = await reason(
    log + "Did the build succeed?",
    {"success": False, "reason": ""}
)
if analysis["data"]["success"]:
    await act("deploy", "deploy to production")
else:
    await act("notify", analysis["data"]["reason"])
```

This script uses Python's native control flow to branch between deployment and notification based on the AI's structured analysis.

### 4.2 Sandbox

Two sandbox strategies are supported:

- **Cloud Sandbox** — the RAS runs in a managed cloud environment (e.g. E2B, Deno Sandbox, Modal, Daytona). Recommended for production agent deployments.
- **Client-side Sandbox** — the RAS runs in a controlled local or self-hosted environment (e.g. Docker, WASM, Cloudflare Workers).

## 5. Interfaces

`reason()` **MUST** be supported regardless of the RAS chosen. `act()` is **OPTIONAL** and may be substituted by any user-defined action execution strategy. When both are provided, they share the same contract across Bash RAS and Python RAS.

### 5.1 `reason([prompt], [example_output])`

**Description:** A local regulator interface that maps non-deterministic language inputs into deterministic structured JSON for execution control. In control-theoretic terms, `reason()` converts local observations plus a reference objective into a bounded control signal. It is an atomic reasoning step — no tools, no memory, no side effects.

**Control-theoretic contract:**

1. **Reference + observation in, control signal out** — prompts should contain the target objective (reference), relevant local state/feedback (observation), and constraints.
2. **Bounded variety output** — `example_output` defines the admissible response shape, constraining output variety into an executable channel.
3. **Isolated local regulation** — `reason()` must not call tools or depend on hidden conversation state; otherwise the regulation boundary becomes ambiguous.
4. **Explicit failure semantics** — on repeated schema failure, return structured error output rather than unconstrained text.

**Algorithm:**

1. Infer a JSON schema from `example_output`
2. Request the LLM for structured output using the prompt and inferred schema
3. Validate the structured output against the schema
   - If validation **fails**: retry up to N−1 times with the validation error as feedback; on the N-th failure, return `{ "error": "<error message>" }`
   - If validation **passes**: return `{ "data": <structured output> }`

**Implementation options:**

- **New conversation context** — reuse your LLM request logic but start a fresh context (similar to the "subagent" approach)
- **MCP Sampling** — where the RAS acts as an MCP Server and requests completion from the MCP client

### 5.2 `act([name], [args])` — Optional

**Description:** Gets external resources or calls external tools (MCP tools, custom functions, etc.).

**Definition:** Given a tool name and its proposed arguments, call the tool and return its result. On failure, return the error for better feedback.

> **Note:** Action execution is ultimately up to the runtime and user. `act()` is provided as a standard reference interface. Implementations may substitute any user-defined action execution strategy.
